## Codebase Patterns
- Monorepo uses pnpm workspaces with packages in `packages/*`
- Package naming: `@typokit/{package-name}`, version `0.1.0`, all `"private": true`
- Nx handles build orchestration: `nx run-many --target={build|test|lint|typecheck}`
- ESM modules throughout (`"type": "module"`)
- Build scripts: each package uses `pnpm approve-builds` for native deps (esbuild, sharp)
- `.astro/` is gitignored; generated types are auto-created by `astro sync` / `astro build`
- Starlight social config uses object format `{ github: "url" }`, NOT array format
- Docs package needs `project.json` to override Nx target defaults (uses astro commands, not tsc)
- Starlight sidebar slugs reference paths under `src/content/docs/` without extension; `index.mdx` uses directory path as slug
- GitHub Pages for project repos need `base: "/repo-name"` in astro.config.mjs
- CI workflow uses `pnpm nx build docs` to build the docs package (not `nx run docs:build`)
- starlight-typedoc uses `entryPointStrategy: "packages"` for monorepo; each package dir is an entry point
- Generated API reference files go to `api-reference/generated/` and are gitignored (regenerated on build)
- Use `typeDocSidebarGroup` from starlight-typedoc to insert auto-generated sidebar; replaces manual API Reference section
- Set `errorOnEmptyDocumentation: false` since some packages may lack full JSDoc coverage
- Starlight UI components (Tabs, Steps, Aside, etc.) import from `@astrojs/starlight/components` in MDX files
- Internal doc links require the `base` prefix for GitHub Pages (e.g., `/typokit/path/to/page/`)
