## Codebase Patterns
- `JSON_HEADERS` from `@typokit/core` is a frozen `{ "content-type": "application/json" }` constant ŌĆö always use it instead of inline header objects in handlers
- `shared-routes.ts` is the Node handler file; `shared-routes-bun.ts` is the Bun variant using `bun:sqlite`; `shared-routes-common.ts` has only route table/validators (no handlers)
- Nx caches builds ŌĆö typecheck may say "read from cache" if build already verified types
- Both `normalizeRequest()` (platform-node and platform-bun) return fresh object literals per request ŌĆö safe to mutate `req.params` directly in server-native
- Guard `Object.assign(ctx, added)` with `if (added != null && typeof added === 'object' && hasOwnKeys(added))` ŌĆö skip for null/undefined/empty returns
- `MiddlewareChain.entries` (from @typokit/types) have no `priority` field ŌĆö sorting is unnecessary when compiling from framework entries directly
- server-native compiles middleware directly from MiddlewareFn handlers (req, ctx, next) ŌĆö no MiddlewareInput adaptation layer
- `createRequestContext()` uses `Object.create(baseContext)` ŌĆö `log` and `fail` are on the prototype (shared), only `requestId` and `services` are per-request own properties
- server-native detects handler.length >= 2 at registration time to skip context allocation for handlers that don't use ctx
- `MiddlewareEntry.noOp` and `MiddlewareChain` entry `noOp` flag filter out no-op middleware at compile time ŌĆö `compileMiddlewareChain()` returns sync identity when all entries are noOp
- `CompiledMiddlewareFn` returns `RequestContext | Promise<RequestContext>` ŌĆö use `instanceof Promise` check at call site to avoid unnecessary microtask overhead
- platform-bun `copyHeaders()` eagerly copies Web API Headers via `forEach()` into a plain object ŌĆö no Proxy wrapper. Future stories (US-022, US-023, US-028) build on this.
- platform-bun files may have CRLF line endings ŌĆö run `eslint --fix` to normalize before committing
