## Codebase Patterns
- `JSON_HEADERS` from `@typokit/core` is a frozen `{ "content-type": "application/json" }` constant — always use it instead of inline header objects in handlers
- `shared-routes.ts` is the Node handler file; `shared-routes-bun.ts` is the Bun variant using `bun:sqlite`; `shared-routes-common.ts` has only route table/validators (no handlers)
- Nx caches builds — typecheck may say "read from cache" if build already verified types
- Both `normalizeRequest()` (platform-node and platform-bun) return fresh object literals per request — safe to mutate `req.params` directly in server-native
- Guard `Object.assign(ctx, added)` with `if (added != null && typeof added === 'object' && hasOwnKeys(added))` — skip for null/undefined/empty returns
- `MiddlewareChain.entries` (from @typokit/types) have no `priority` field — sorting is unnecessary when compiling from framework entries directly
- server-native compiles middleware directly from MiddlewareFn handlers (req, ctx, next) — no MiddlewareInput adaptation layer
- `createRequestContext()` uses `Object.create(baseContext)` — `log` and `fail` are on the prototype (shared), only `requestId` and `services` are per-request own properties
- server-native detects handler.length >= 2 at registration time to skip context allocation for handlers that don't use ctx
- `MiddlewareEntry.noOp` and `MiddlewareChain` entry `noOp` flag filter out no-op middleware at compile time — `compileMiddlewareChain()` returns sync identity when all entries are noOp
- `CompiledMiddlewareFn` returns `RequestContext | Promise<RequestContext>` — use `instanceof Promise` check at call site to avoid unnecessary microtask overhead
- platform-bun `copyHeaders()` eagerly copies Web API Headers via `forEach()` into a plain object — no Proxy wrapper. Future stories (US-022, US-023, US-028) build on this.
- platform-bun files may have CRLF line endings — run `eslint --fix` to normalize before committing
- platform-bun `env.d.ts` declares `Headers()` with no-arg constructor only — use `h.set()` after construction to populate
- `buildResponse()` uses pre-allocated `JSON_RESPONSE_INIT_{200,400,404,500}` with plain `JSON_CT_HEADERS` object (not `new Headers()`) — Bun's Response constructor accepts plain objects
- `JSON_CT_HEADERS` is a shared module-level `{ 'content-type': 'application/json' }` plain object — reuse it for any Bun Response construction needing JSON headers
- platform-bun exports `normalizeRequestSync` (GET/HEAD/DELETE/OPTIONS) and `normalizeRequestAsync` (POST/PUT/PATCH) — use directly for optimal perf; `normalizeRequest` is a backward-compat wrapper
- `BODYLESS_METHODS` Set in platform-bun: GET, HEAD, DELETE, OPTIONS — use `BODYLESS_METHODS.has(method)` to branch sync/async
- `extractPathAndQuery()` is the shared URL parsing helper in platform-bun — reused by both sync and async variants
- `JSON_CT_HEADERS` is exported from platform-bun as `Record<string, string>` — use for Bun Response construction headers
- `createBunServer` accepts `BunServerOptions.fastPath` callback `(req: Request) => Response | Promise<Response> | null` — bypass normalize/build for simple routes
- server-native fast-path uses pre-allocated `fpJsonInit{200,404,500}` ResponseInit with plain header objects for zero-allocation Bun Response construction
- server-native on Bun calls `Bun.serve()` directly in `listen()` — no `createBunServer` wrapper. Import `normalizeRequestSync`, `normalizeRequestAsync`, `buildResponse` from platform-bun for the normal path.
