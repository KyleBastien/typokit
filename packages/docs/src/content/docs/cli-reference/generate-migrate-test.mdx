Reference for TypoKit CLI commands that generate code from your schema, manage database migrations, and run tests.

Every command accepts these **global flags**:

| Flag | Description |
|------|-------------|
| `--root <dir>` | Project root directory (default: current working directory) |
| `--verbose`, `-v` | Show detailed output including timing and file-level information |

## Generate Commands

All generate commands read your TypeScript type files (configured via `typeFiles` in `typokit.config.ts`) and route files, then emit derived artifacts to the output directory (default: `.typokit/`).

### `typokit generate:db`

Generate database schema files (SQL DDL and JSON metadata) from your TypeScript entity types.

**Syntax**

```bash
typokit generate:db
```

**Flags**

| Flag | Description |
|------|-------------|
| `--verbose`, `-v` | Show detailed output |
| `--json` | Output result metadata as JSON |

**What it does**

1. Reads all type files from the configured `typeFiles` globs
2. Extracts entity interfaces annotated with `@table` JSDoc tags
3. Calls `diffSchemas()` to compare current types against the last-known schema snapshot
4. Emits SQL DDL statements and a JSON metadata file to the output directory

**Example**

```bash
typokit generate:db
```

```
â—Ź Reading type files...        12 files
â—Ź Extracting entities...       6 tables
â—Ź Generating DDL...            done

âś” Database schema generated (6 tables)
  .typokit/db/schema.sql       (2.4kB)
  .typokit/db/schema.json      (1.1kB)
```

With `--json`:

```bash
typokit generate:db --json
```

```json
{
  "success": true,
  "filesWritten": [".typokit/db/schema.sql", ".typokit/db/schema.json"],
  "tables": 6,
  "duration": 312
}
```

<Aside type="tip">
The generated DDL is a complete snapshot of your schema, not a migration. Use `typokit migrate:generate` to create incremental migration files.
</Aside>

### `typokit generate:client`

Generate a type-safe API client from your route contracts.

**Syntax**

```bash
typokit generate:client
```

**Flags**

| Flag | Description |
|------|-------------|
| `--verbose`, `-v` | Show detailed output |
| `--json` | Output result metadata as JSON |

**What it does**

1. Reads all route files from the configured `routeFiles` globs
2. Extracts route contracts with their typed parameters, query strings, request bodies, and responses
3. Generates a TypeScript client module with a `createClient()` factory function
4. Emits the client to the output directory

**Example**

```bash
typokit generate:client
```

```
â—Ź Reading route files...       8 modules
â—Ź Extracting contracts...      24 operations
â—Ź Generating client...         done

âś” Client generated (24 operations)
  .typokit/client/index.ts     (6.8kB)
```

Generated client usage:

```typescript
import { createClient } from ".typokit/client";

const api = createClient({ baseUrl: "http://localhost:3000" });

// Fully typed â€” params, query, body, and response
const users = await api.getUsers({ query: { page: 1 } });
const user = await api.getUser({ params: { id: "abc" } });
```

<Aside type="note">
The generated client is deterministic â€” the same route contracts always produce identical client code. Regenerating after changes is safe and idempotent.
</Aside>

### `typokit generate:openapi`

Generate an OpenAPI 3.1 specification from your routes and types.

**Syntax**

```bash
typokit generate:openapi [--output <path>]
```

**Flags**

| Flag | Description |
|------|-------------|
| `--output <path>` | Custom output path (default: `.typokit/openapi.json`) |
| `--verbose`, `-v` | Show detailed output |
| `--json` | Output result metadata as JSON |

**What it does**

1. Reads all route files and type files
2. Builds a complete OpenAPI 3.1 specification including paths, schemas, parameters, and responses
3. Writes the spec as a JSON file to the output location

**Example**

```bash
typokit generate:openapi
```

```
â—Ź Reading routes and types...  8 modules, 12 type files
â—Ź Building OpenAPI spec...     24 operations, 18 schemas
â—Ź Writing spec...              done

âś” OpenAPI spec generated
  .typokit/openapi.json        (14.2kB)
```

With a custom output path:

```bash
typokit generate:openapi --output ./dist/openapi.json
```

```
âś” OpenAPI spec generated
  dist/openapi.json             (14.2kB)
```

<Aside type="tip">
Use the `--output` flag to place the spec in a location accessible by tools like Swagger UI, Redoc, or CI pipelines that validate API contracts.
</Aside>

## Migrate Commands

Migration commands manage incremental database schema changes. Migrations are always generated as drafts for review â€” they never auto-apply.

### `typokit migrate:generate`

Generate a migration draft from the diff between your current TypeScript types and the last-known schema snapshot.

**Syntax**

```bash
typokit migrate:generate --name <name>
```

**Flags**

| Flag | Description |
|------|-------------|
| `--name <name>` | Migration name (required) |
| `--verbose`, `-v` | Show detailed output |
| `--json` | Output result metadata as JSON |

**What it does**

1. Compares the current entity types against the saved schema snapshot
2. Generates a timestamped SQL migration file with the detected changes
3. Generates a JSON metadata file describing each change
4. Annotates destructive changes (column drops, type changes) with `-- DESTRUCTIVE: requires review`
5. Updates the schema snapshot after generation

**Example**

```bash
typokit migrate:generate --name add-user-avatar
```

```
â—Ź Comparing schemas...         2 changes detected
â—Ź Generating migration...      done

âś” Migration generated
  .typokit/migrations/20260302_120000_add-user-avatar.sql
  .typokit/migrations/20260302_120000_add-user-avatar.json

Changes:
  + users.avatar_url       VARCHAR(255)    (add column)
  + users.avatar_updated   TIMESTAMP       (add column)
```

When destructive changes are detected:

```bash
typokit migrate:generate --name remove-legacy-fields
```

```
â—Ź Comparing schemas...         3 changes detected (1 destructive)
â—Ź Generating migration...      done

âś” Migration generated
  .typokit/migrations/20260302_130000_remove-legacy-fields.sql
  .typokit/migrations/20260302_130000_remove-legacy-fields.json

Changes:
  - users.legacy_role      VARCHAR(50)     (drop column) âš  DESTRUCTIVE
  ~ posts.status           ENUM â†’ VARCHAR  (type change)
  + posts.published_at     TIMESTAMP       (add column)
```

<Aside type="caution">
Destructive changes are annotated with `-- DESTRUCTIVE: requires review` in the SQL file. Always review generated migration files before applying them.
</Aside>

### `typokit migrate:apply`

Apply pending migrations to the database.

**Syntax**

```bash
typokit migrate:apply [--force]
```

**Flags**

| Flag | Description |
|------|-------------|
| `--force` | Apply destructive migrations without interactive review |
| `--verbose`, `-v` | Show detailed output |
| `--json` | Output result metadata as JSON |

**What it does**

1. Reads all migration files from the migrations directory
2. Checks which migrations have already been applied (tracked in `.applied` log)
3. Applies pending migrations in chronological order
4. Blocks on destructive migrations unless `--force` is passed
5. Records applied migrations in the `.applied` log

**Example**

```bash
typokit migrate:apply
```

```
â—Ź Checking pending migrations...  2 pending
â—Ź Applying 20260302_120000_add-user-avatar.sql...    âś”
â—Ź Applying 20260302_130000_add-indexes.sql...         âś”

âś” 2 migrations applied (148ms)
```

When destructive migrations are pending:

```bash
typokit migrate:apply
```

```
â—Ź Checking pending migrations...  1 pending

âš  Migration 20260302_130000_remove-legacy-fields.sql contains destructive changes:
  - users.legacy_role (drop column)

Blocked. Use --force to apply destructive migrations.
```

```bash
typokit migrate:apply --force
```

```
â—Ź Checking pending migrations...  1 pending
â—Ź Applying 20260302_130000_remove-legacy-fields.sql... âś” (destructive)

âś” 1 migration applied (92ms)
```

With `--json`:

```bash
typokit migrate:apply --json
```

```json
{
  "success": true,
  "applied": [
    {
      "file": "20260302_120000_add-user-avatar.sql",
      "duration": 84,
      "destructive": false
    }
  ],
  "total": 1,
  "duration": 148
}
```

<Aside type="caution">
Destructive migrations (column drops, type changes) are blocked by default. Always review the SQL file before using `--force`. In CI pipelines, use `migrate:diff --json` to check for destructive changes programmatically before applying.
</Aside>

## Test Commands

Test commands run your project's tests using auto-detected or explicitly specified test runners.

TypoKit detects the test runner by checking for config files in this priority order:

| Priority | Runner | Config files checked |
|----------|--------|---------------------|
| 1 | Jest | `jest.config.{js,ts,mjs,cjs}` |
| 2 | Vitest | `vitest.config.{js,ts,mjs,cjs}` |
| 3 | Rstest | `rstest.config.{js,ts,mjs,cjs}` |
| fallback | Vitest | (used when no config found) |

### `typokit test`

Run all tests (contract tests + integration tests) using the detected test runner.

**Syntax**

```bash
typokit test [--runner <runner>]
```

**Flags**

| Flag | Description |
|------|-------------|
| `--runner <runner>` | Override test runner detection (`jest`, `vitest`, or `rstest`) |
| `--verbose`, `-v` | Show verbose test output |
| `--json` | Output result metadata as JSON |

**What it does**

1. Auto-detects the test runner (or uses the `--runner` override)
2. Checks if contract test stubs are stale â€” regenerates them if the schema has changed since the last build
3. Spawns the test runner as a subprocess with the full test suite

**Example**

```bash
typokit test
```

```
â—Ź Detected runner: vitest
â—Ź Checking contract tests...   up to date
â—Ź Running tests...

 âś“ __generated__/tests/getUsers.contract.test.ts (3 tests)

 âś“ tests/integration/auth.test.ts (4 tests)

âś” All tests passed
```

With `--runner` override:

When contract tests are regenerated automatically, the output includes a note: `regenerated (schema changed)`. This means your route contracts changed since the last build and fresh test stubs were generated before running.

### `typokit test:contracts`

Run only the generated contract tests.

**Syntax**

```bash
typokit test:contracts [--runner <runner>]
```

**Flags**

| Flag | Description |
|------|-------------|
| `--runner <runner>` | Override test runner detection (`jest`, `vitest`, or `rstest`) |
| `--verbose`, `-v` | Show verbose test output |
| `--json` | Output result metadata as JSON |

**What it does**

1. Auto-detects the test runner (or uses the `--runner` override)
2. Regenerates contract tests if schemas have changed
3. Runs only the tests in `__generated__/tests/` using runner-specific path patterns

Runner-specific patterns:

| Runner | Pattern |
|--------|---------|
| Jest | `--testPathPattern __generated__/.*\.contract\.test` |
| Vitest | `run __generated__/` |
| Rstest | `run --testPathPattern __generated__/.*\.contract\.test` |

**Example**

```bash
typokit test:contracts
```

```
â—Ź Detected runner: vitest
â—Ź Checking contract tests...   up to date
â—Ź Running contract tests...

 âś“ __generated__/tests/getUsers.contract.test.ts (3 tests)
 âś“ __generated__/tests/createUser.contract.test.ts (2 tests)
 âś“ __generated__/tests/getUser.contract.test.ts (2 tests)
 âś“ __generated__/tests/updateUser.contract.test.ts (2 tests)
 âś“ __generated__/tests/deleteUser.contract.test.ts (1 test)
```

<Aside type="tip">
Contract tests validate that your API responses match the shapes defined in your route contracts. Run them after changing type definitions to catch schema mismatches early.
